title: Generated code

This page shows how Protobuf-ES represents Protobuf definitions in generated JavaScript and TypeScript.

Unless otherwise noted, the examples use this schema:

```protobuf
syntax = "proto3";
package example;

message User {
  string first_name = 1;
  string last_name = 2;
  bool active = 3;
  User manager = 4;
  repeated string locations = 5;
  map<string, string> projects = 6;
}
```

## Files

For every `.proto` file, the plugin generates a corresponding `.js`, `.ts`, or `.d.ts` file with the suffix `_pb`.

For example, `foo/bar.proto` becomes `foo/bar_pb.js`.

Each generated file starts with a short preamble:

```typescript
// @generated by protoc-gen-es v2.0.0 with parameter "target=dts"
// @generated from file example.proto (package example, syntax proto3)
/* eslint-disable */
```

Generated imports are relative and follow the options you configured. If your Protobuf file imports another file, the generated code imports the corresponding `_pb` output.

Below the imports, Protobuf-ES generates the file schema:

```typescript
/**
 * Describes the file example.proto.
 */
export declare const file_example: GenFile;
```

You usually need this for registries and reflection.

## Messages

For the `User` message above, Protobuf-ES generates a message type and its schema:

```typescript
import type { Message } from "@bufbuild/protobuf";

/**
 * @generated from message example.User
 */
export declare type User = Message<"example.User"> & {
  /**
   * @generated from field: string first_name = 1;
   */
  firstName: string;
};

export declare const UserSchema: GenMessage<User>;
```

If you have used libraries like Zod, the shape is familiar. The schema is the typed description of the message, and you pass it to runtime helpers such as `create()`, `fromBinary()`, and `toBinary()`.

```typescript
import { fromBinary } from "@bufbuild/protobuf";
import { UserSchema } from "./gen/example_pb";

const bytes = new Uint8Array([10, 3, 84, 105, 109]);
fromBinary(UserSchema, bytes); // User
```

## Scalar fields

Scalar fields are generated as the closest matching ECMAScript type.

| Protobuf type | ECMAScript type | Notes | Default value |
|---|---|---|---|
| `string` | `string` | UTF-8 | `""` |
| `bool` | `boolean` |  | `false` |
| `bytes` | `Uint8Array` |  | `new Uint8Array(0)` |
| `double` | `number` | 64-bit float | `0` |
| `float` | `number` | 32-bit float | `0` |
| `int32` | `number` | 32-bit signed varint | `0` |
| `uint32` | `number` | 32-bit unsigned varint | `0` |
| `int64` | `bigint` | 64-bit signed varint | `0n` |
| `uint64` | `bigint` | 64-bit unsigned varint | `0n` |
| `fixed32` | `number` | 32-bit unsigned fixed | `0` |
| `fixed64` | `bigint` | 64-bit unsigned fixed | `0n` |
| `sfixed32` | `number` | 32-bit signed fixed | `0` |
| `sfixed64` | `bigint` | 64-bit signed fixed | `0n` |
| `sint32` | `number` | 32-bit signed varint | `0` |
| `sint64` | `bigint` | 64-bit signed varint | `0n` |

If `bigint` is unavailable in your environment, 64-bit integer fields still round-trip safely, but use `string` values at runtime instead.

If you want `string` output for a 64-bit field, use `jstype = JS_STRING`:

```protobuf
int64 field = 1 [jstype = JS_STRING];
```

With [Buf managed mode](https://buf.build/docs/generate/managed-mode#optimization-and-field-options), you can apply that option automatically.

## Message fields

Message fields become optional properties:

```protobuf
User manager = 4;
```

```typescript
/**
 * @generated from field: example.User manager = 4;
 */
manager?: User;
```

Message fields do not have default values in Protobuf.

## Enum fields

Enum fields use the generated TypeScript enum:

```protobuf
PhoneType phone_type = 3;
```

```typescript
/**
 * @generated from field: example.PhoneType phone_type = 3;
 */
phoneType: PhoneType;
```

Enum fields default to the first declared value.

## Repeated fields

Repeated fields become arrays:

```protobuf
repeated string locations = 5;
```

```typescript
/**
 * @generated from field: repeated string locations = 5;
 */
locations: string[] = [];
```

## Map fields

Map fields become record-like objects:

```protobuf
map<string, string> projects = 6;
```

```typescript
/**
 * @generated from field: map<string, string> projects = 6;
 */
projects: { [key: string]: string } = {};
```

Protobuf-ES deliberately uses objects instead of ECMAScript `Map` because objects fit better with current JavaScript and TypeScript tooling.

## Oneof fields

Oneofs become discriminated unions.

```protobuf
oneof result {
  int32 number = 1;
  string error = 2;
}
```

```typescript
result:
  | { case: "number"; value: number }
  | { case: "error"; value: string }
  | { case: undefined; value?: undefined } = { case: undefined };
```

Use normal control flow to narrow the type:

```typescript
switch (user.result.case) {
  case "number":
    user.result.value; // number
    break;
  case "error":
    user.result.value; // string
    break;
}
```

This representation works especially well with TypeScript's narrowing rules.

## Proto2 group fields

Groups are deprecated in proto2, but Protobuf-ES supports them.

```protobuf
optional group MyGroup = 1 {
  optional int32 int32_field = 1;
}
```

This produces a property and a generated nested message type such as `User_MyGroup`.

## Proto2 required fields

Proto2 `required` fields are validated when serializing a message, not while constructing or parsing one.

With Protobuf-ES v2, scalar and enum `required` fields are no longer generated as optional properties. Message fields keep their previous behavior.

> [!CAUTION]
>
> `required` is a legacy feature. The official Protobuf language guide says not to use it in new schemas.

## Proto3 optional fields

Proto3 `optional` fields enable presence tracking for scalar and enum fields.

```protobuf
optional bool active = 3;
```

```typescript
/**
 * @generated from field: optional bool active = 3;
 */
active?: boolean;
```

See [Working with messages](/working-with-messages/) for presence tracking at runtime.

## Field names

Generated property names are always `lowerCamelCase`, even if the Protobuf source uses `snake_case`.

Protobuf-ES follows Protobuf's standard JSON name conversion rules and escapes reserved words when necessary.

## Enumerations

Enumerations become TypeScript enums.

```protobuf
enum PhoneType {
  PHONE_TYPE_UNSPECIFIED = 0;
  PHONE_TYPE_MOBILE = 1;
  PHONE_TYPE_LAND_LINE = 2;
}
```

```typescript
export enum PhoneType {
  UNSPECIFIED = 0,
  MOBILE = 1,
  LAND_LINE = 2,
}
```

If every enum value shares a prefix that matches the enum name, Protobuf-ES drops the prefix in generated code.

The enum schema is generated alongside the enum values:

```typescript
export declare const PhoneTypeSchema: GenEnum<PhoneType>;
```

## Extensions

Extensions become typed extension descriptors:

```protobuf
syntax = "proto2";

message User {
  extensions 100 to 200;
}

extend User {
  optional uint32 age = 100;
}
```

```typescript
export declare const age: GenExtension<User, number>;
```

At runtime, use helpers such as `setExtension()`, `getExtension()`, `hasExtension()`, and `clearExtension()`.

See [Reflection](/reflection/) for registries and custom options built on top of the same descriptor model.

## Services

For services, Protobuf-ES generates schemas only:

```protobuf
service UserService {
  rpc CreateUser(CreateUserRequest) returns (CreateUserResponse);
}
```

```typescript
export declare const UserService: GenService<{
  createUser: {
    methodKind: "unary";
    input: typeof CreateUserRequestSchema;
    output: typeof CreateUserResponseSchema;
  };
}>;
```

Protobuf-ES does not implement RPC transports itself. Libraries such as [Connect-ES](https://github.com/connectrpc/connect-es) use these typed service schemas.

## Reserved names

Some names that are legal in Protobuf are reserved in ECMAScript or conflict with built-in properties. Protobuf-ES escapes them by appending `$`.

## Names for nested types

Nested message and enum names are joined with underscores, similar to the Go implementation.

For example, the nested enum `User.Type` becomes `User_Type` in generated code.

## Comments

Comments in `.proto` files are carried over as JSDoc comments in generated output, including license headers, field comments, service comments, and deprecation markers.

```typescript
/**
 * This field is deprecated
 *
 * @generated from field: string deprecated_field = 1 [deprecated = true];
 * @deprecated
 */
deprecatedField = "";
```

## Packages

ECMAScript does not have Protobuf-style namespaces, so package declarations are mostly ignored in generated identifiers. They remain visible in descriptors and fully qualified type names.
